# Конспект по BPMN 2.0

## 1. Назначение BPMN

BPMN 2.0 — это нотация для графического описания бизнес-процессов. Она используется для того, чтобы показать последовательность действий, участников процесса, условия ветвления, события и варианты завершения процесса. BPMN отличается от обычной блок-схемы тем, что каждый элемент имеет строгое значение и читается по установленным правилам. Такая нотация удобна для бизнес-аналитиков, разработчиков, руководителей и других специалистов, которым нужно одинаково понимать процесс. В рамках практики BPMN применяется для моделирования выбранного процесса покупки цифровой игры в Steam.

| Start Event           | Круг с тонкой границей                                     | Начало процесса                                              | Пользователь начинает оформление покупки игры                                |
| End Event             | Круг с толстой границей                                    | Завершение процесса                                          | Игра добавлена в библиотеку пользователя                                     |
| Intermediate Event    | Круг с двойной границей                                    | Событие, возникающее в середине процесса                     | Ожидание подтверждения платежа                                               |
| Task                  | Прямоугольник со скруглёнными углами                       | Отдельное действие в процессе                                | Выбрать игру, добавить игру в корзину, подтвердить покупку                   |

| Pool                  | Большая область, обозначающая участника процесса           | Отдельный участник или организация                           | Пользователь, Steam, платёжный сервис                                        |

## 3. Типы шлюзов BPMN

| Тип шлюза           | Условие применения                                                                                   | Пример из практики                                                                                              | Типичная ошибка применения                                                         |
| ------------------- | ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------------------------------------------- | ---------------------------------------------------------------------------------- |
| XOR Gateway         | Используется, когда из нескольких вариантов может быть выбран только один                            | После проверки оплаты процесс идёт либо по ветке «оплата успешна», либо по ветке «оплата отклонена»             | Не подписывать условия на исходящих потоках                                        |
| AND Gateway         | Используется, когда нужно запустить несколько действий параллельно и дождаться завершения всех веток | После успешной покупки Steam может одновременно добавить игру в библиотеку и отправить уведомление пользователю | Использовать AND вместо XOR, когда на самом деле должен быть выбор одного варианта |
| OR Gateway          | Используется, когда может выполниться одна или несколько веток одновременно                          | Пользователь может получить уведомление в приложении, по электронной почте или обоими способами                 | Использовать OR там, где нужен простой исключающий выбор                           |
| Event-based Gateway | Используется, когда дальнейший путь зависит от наступившего события                                  | После запроса оплаты процесс ждёт либо подтверждение платежа, либо сообщение об ошибке                          | Путать событие с обычной задачей или использовать без реального ожидания события   |

## 4. Типы подпроцессов

### Collapsed Sub-Process

Collapsed Sub-Process — это свёрнутый подпроцесс. На основной диаграмме он выглядит как один блок с маркером «+», но его внутреннее содержание раскрывается на отдельном листе или отдельной диаграмме. Такой вариант используется, когда основная диаграмма становится слишком большой или когда нужно показать детали процесса отдельно.

Пример для Steam: на основной диаграмме можно показать подпроцесс **«Провести оплату»**, а на отдельной диаграмме раскрыть его шаги: передать данные платежа, проверить платёж, получить результат, вернуть статус операции в Steam.

### Expanded Sub-Process

Expanded Sub-Process — это раскрытый подпроцесс, который показывается прямо внутри основной диаграммы. Внутри такого блока видны задачи, события и потоки. Его удобно применять, когда подпроцесс небольшой и его детализация не перегружает схему.

Пример для Steam: если проверка заказа состоит всего из нескольких действий, её можно показать прямо внутри основной диаграммы: проверить аккаунт, проверить доступность игры, сформировать сумму заказа.

### Call Activity

Call Activity — это вызов повторно используемого процесса. Он применяется, если один и тот же процесс может использоваться в разных местах. Например, общий процесс **«Обработать платёж»** может использоваться не только при покупке игры, но и при покупке дополнения, пополнении кошелька Steam или оплате внутриигрового товара.

## 5. Авторский чек-лист проверки BPMN-диаграммы

1. У диаграммы есть понятное начало процесса — Start Event.
2. Каждая завершающая ветка процесса заканчивается End Event.
3. Все задачи названы глагольными формами, например: «Выбрать игру», «Провести оплату», «Добавить игру в библиотеку».
4. Задачи расположены в дорожках тех участников, которые реально их выполняют.
5. Sequence Flow используется только внутри одного пула и не пересекает границы пулов.
6. Message Flow используется только между разными пулами.
7. У каждого XOR-шлюза подписаны условия на исходящих потоках.
8. Если используется AND-шлюз для разделения, должен быть соответствующий AND-шлюз для объединения параллельных веток.
9. Подпроцесс имеет более высокий уровень обобщения, чем обычная задача, и не дублирует её название.
10. У свёрнутого подпроцесса есть отдельная раскрытая диаграмма.
11. На диаграмме нет «висящих» стрелок: каждый поток начинается и заканчивается конкретным элементом.
12. Диаграмма не перегружена: если процесс становится слишком большим, часть логики лучше вынести в подпроцесс.
13. Названия элементов понятны без дополнительных объяснений.
14. Логика процесса соответствует реальному сценарию покупки цифровой игры в Steam.
15. Ошибочные или альтернативные сценарии, например ошибка оплаты, показаны отдельной веткой процесса.

## 6. Применение BPMN к процессу Steam

В дальнейшей работе BPMN будет применяться для моделирования процесса **«Обработать покупку цифровой игры в Steam»**. Основными участниками процесса будут пользователь, платформа Steam, платёжный сервис и библиотека Steam. В качестве основного сценария будет рассмотрена успешная покупка игры, а в качестве альтернативного сценария — ошибка оплаты или отмена покупки.

Предполагаемые задачи основного процесса:

1. Выбрать игру в магазине Steam.
2. Добавить игру в корзину.
3. Выбрать способ оплаты.
4. Подтвердить покупку.
5. Передать данные платежа в платёжный сервис.
6. Проверить результат оплаты.
7. Добавить игру в библиотеку пользователя.
8. Отправить пользователю уведомление о результате операции.

В итоговой диаграмме можно выделить два подпроцесса: **«Провести оплату»** и **«Добавить игру в библиотеку»**. Первый подпроцесс будет связан с проверкой и подтверждением платежа, второй — с выдачей цифровой лицензии и обновлением библиотеки пользователя.
